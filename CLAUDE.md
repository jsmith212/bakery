# Bakery

A multi-tenant build cache server: Yocto (sstate + hash equivalence + source premirror), Bazel-protocol remote cache (for moon), ccache/sccache, and a Docker pull-through proxy. One Go binary + Postgres.

## Read these first

- `docs/design/DESIGN.md` — the approved architecture and milestone plan. **Start here.**
- `docs/design/protocols/yocto.md` — wire-level sstate / hashserv / premirror reference.
- `docs/design/protocols/bazel-ccache-docker.md` — wire-level REAPI / ccache / OCI reference.
- `docs/design/protocols/client-config.md` — verified client config for every supported tool. Source of truth for the UI's snippet generator.

The protocol docs were built by reading upstream **client source**, not docs. Several findings contradict the obvious assumption. Do not "simplify" against them without re-verifying against source.

## Stack & conventions

Modeled on `../kbi` (adapt, don't copy blindly — kbi uses templ/htmx and go-envconfig; we use SvelteKit and Kong).

- **Go, stdlib-first.** `net/http` `ServeMux` (Go 1.22 method+pattern routing), `log/slog`. No chi/echo/gorilla.
- **Kong** (`alecthomas/kong`) for CLI + env binding. Replaces kbi's `sethvargo/go-envconfig`.
- **pgx/v5 + pgxpool** directly. Not `database/sql`.
- **sqlc** → `internal/db/repository/` (generated, gitignored). Schema is read from the migrations dir; no separate schema.sql.
- **golang-migrate**, migrations embedded via `go:embed` and applied at boot.
- **`go tool` directives** in go.mod pin codegen binaries (`go tool sqlc generate`). Modern replacement for `tools.go`.
- **SvelteKit** → static build → `//go:embed all:dist`. **The `all:` prefix is mandatory** — plain `//go:embed dist` silently skips SvelteKit's `_app/` directory and you ship a white page.
- **Tests:** table-driven, colocated, same-package, stdlib only. No testify, no gomock — hand-written fakes in `mocks_test.go`, matching kbi. Frontend: Vitest + Playwright.
- **Just** is the single source of truth; CI is a thin wrapper over it. Every recipe has a `#` doc comment.
- Conventional commits (commitizen), pre-commit hooks, `just check` = `check-format vet lint web-check`.
- **`internal/db/repository/` is generated and gitignored.** A fresh clone does not compile until `just generate` (`go tool sqlc`) has run. Every recipe, CI job and Docker stage that compiles depends on it. Bootstrap is: `npm ci` (in `web/`) → `just generate` → `go build ./...`, or just `just build`, which does all three.
- **`just test-db` runs the whole Go suite and fails if any test SKIPPED.** `dbtest` skips — it does not fail — when it can find neither docker nor `TEST_DB_URL`, which is right on a laptop and catastrophic in CI. This is the recipe CI runs; a suite that silently skips is not a passing suite.

## Backend: what exists (M1, M1.5 and M2 have landed)

`internal/db` (pool + embedded migrations + boot lock + a `Store` whose `Tx` really rebinds sqlc's `Queries` onto the transaction; the pool's `AfterConnect` **registers every custom enum and its array type** — without that, pgx cannot build an encode plan for an enum-array parameter and the query fails at its *first* caller) · `internal/db/dbtest` (a private migrated Postgres per test) · `internal/slug` · `internal/storage` (**local disk only — S3 is deferred**) · `internal/blob` (dedup, refcount, sharded LRU, singleflight, the headline metrics) · `internal/cache` (`Route`/`Deps`/`Backend`: the seams the backends plug into) · `internal/cache/httpblob` (**M2: the shared sstate + downloads handler, per-backend `Policy`, and the in-process `CachedResolver`**) · `internal/metrics` · `internal/config` (Kong + the group map) · `internal/auth` · `internal/api` (`/api/v1`, incl. the config-snippet generator) · `internal/cli` (the binary is also the API client, incl. `bakery {sstate,downloads} push`) · `internal/server`.

`server.Boot` is the wiring, and it is what `bakery serve` runs *and* what the end-to-end test boots — not a lookalike. Boot order is load-bearing: connect and **ping** → advisory boot lock → migrate → metrics → auth → `EnsureOrgs` → `SeedDevLogin` → API → **construct the M2 cache backends** → bind both listeners → serve. `bakery serve --headless` serves the API, metrics **and the cache backends** and simply does not route the SPA — "no console" is not "no cache".

**Roles live on four independent planes** (M1.5, `docs/design/specs/2026-07-12-hybrid-role-model.md`). One group lookup used to answer three questions; it now answers one each:

| Plane | Source of truth |
|---|---|
| Login gate — *may this human use Bakery at all?* | `login_groups` in the group map. **Empty/unset admits any successful OIDC auth.** |
| Site role | Hybrid: `site_admin_groups` **+** local grants (`--allow-local-site-admins`, default on; `bakery user site-admin` is the DB-only break-glass, no HTTP path). |
| Org membership | Hybrid: the group map **+** local grants. Effective role `greatest(oidc_role, local_role)`, generated by the database. |
| Project role | **In-app only.** Never group-mappable — that is the group explosion this design exists to escape. |

The auth model, the full `/api/v1` route table, and what M1/M1.5 did *not* build are recorded in DESIGN.md under "M1 as landed" and "M1.5 as landed". Read it before adding an endpoint.

## Invariants — do not violate these

**Storage ordering.** On delete: metadata first, then bytes. On create: bytes first, then metadata. Orphaned bytes are recoverable; dangling metadata is a permanent 500.

**The HTTP `/ac/` mount is an OPAQUE byte store; the gRPC `ActionCache` is TYPED. They are DIFFERENT `cache_objects.namespace`s (`ac` vs `ac-grpc`) and MUST NOT share one.** `/cache/{o}/{p}/ac/{hex}` (namespace `ac`) round-trips the body verbatim — never parse it, never validate it, never verify `key==sha256(body)`: ccache `@layout=bazel` writes its own binary `CacheEntry` at a padded key that hashes nothing, moon `api:http` writes `Manifest` JSON, Bazel `--remote_cache=http` writes a serialized `ActionResult` — parsing breaks two of the three. gRPC `ActionCache` MUST parse (`GetActionResult` returns a typed `*repb.ActionResult`), so it owns namespace `ac-grpc`, written only by `UpdateActionResult`. It cannot share `ac`: moon computes the SAME action digest over both transports, so one namespace makes an `api:grpc→http` switch read a protobuf where the client wants JSON — a hard client error, not a miss. **`proto.Marshal`-ing the `ActionResult` is NOT the OCI trap** — the AC key is `sha256(Action)`, the value is addressed by nobody (Bazel never digest-verifies AC downloads); the one-question test for "is this the OCI trap?" is *is the value addressed by its OWN bytes?* (manifest yes, ActionResult no, CAS blob yes). Parse-fail on read → `NOT_FOUND` (the one code both clients treat as a clean miss) + a loud metric; NEVER store or serve a zero-length AC value (an empty body `Unmarshal`s into `ActionResult{exit_code:0}` — the only garbage that parses). `/cas/` is content-addressed and MUST verify `key==sha256(body)`; it is safe to share across transports.

**sccache is NOT on `/ac/`; it is WebDAV.** sccache writes to `{endpoint}/{KEY_PREFIX}/{a}/{b}/{c}/{64hex}` (namespace `sccache`, multi-segment `{path...}` route) and its opendal WebDAV client does **PROPFIND → MKCOL → PUT** on every write — a 405 on PROPFIND/MKCOL does not fail loudly, it degrades sccache to `CacheMode::ReadOnly` for the whole process and the cache silently never populates. Answer PROPFIND with `207` declaring the path a collection (`<D:getlastmodified>` RFC-2822 + `<D:resourcetype><D:collection/>`) and MKCOL with `201`.

**REAPI conformance traps (each a silent build-breaker if wrong).** GetCapabilities MUST advertise `action_cache_update_capabilities.update_enabled=true` (else moon uploads every blob and skips `UpdateActionResult` — 0% hit rate, DEBUG log only) and a NON-ZERO `max_batch_total_size_bytes` ≤ 64 MiB (`0`/unset makes moon send ~48 MiB unary messages and stream nothing; 4 MiB is the value, grpc MaxRecv/Send at 16 MiB). ByteStream `committed_size` is always the byte count, never `0`/`-1` on an uncompressed write (Bazel throws "write incomplete"); `QueryWriteStatus` returns `Unimplemented`, NOT the spec's `NOT_FOUND` (which fails Bazel's upload); `GetActionResult` miss is `NotFound` and NOTHING else (every other code is a build-failing `IOException`). Parse ByteStream resource names by scanning for the `blobs`/`uploads`/`compressed-blobs` marker (instance_name has slashes) and OMIT the digest-function segment for sha256 (requiring it rejects 100% of stock traffic). Advertise `IDENTITY` compressors only — advertising ZSTD obliges serving it in two framings; both clients default compression off. The empty blob (`e3b0c442…/0`) is always-present and is special-cased in the BAZEL BACKEND, never in `blob.Service` (which would 500 the sstate mount).

**GC owes M4: sweep `ac`/`ac-grpc` BEFORE `cas`, and never age a `cas` object out on `created_at` alone.** `FindMissingBlobs` means a hot blob is never re-uploaded, so its `created_at` stays pinned at first upload while the `ActionResult` naming it gets a fresh `created_at` every build — an age sweep deletes exactly the old, hot, deduped CAS blobs and keeps the young AC entries pointing at them, and Bazel's response to a swept blob under `--remote_download_minimal` is a build **abort and rewind** (`LostInputsEvent`), not a clean miss. CAS liveness is an access-time property; `cache_objects` has no `accessed_at` yet (an M6 decision). Same shape as "always sweep hashserv before sstate".

**gRPC (REAPI) is served on its OWN `--grpc-addr` listener — NEVER the public port, NEVER `x/net/http2/h2c`.** `grpc.Server.ServeHTTP` (the shared-port demux path) has an unimplemented `Drain`, so `GracefulStop` PANICS the moment an RPC is in flight at shutdown — invisible in tests, certain in production. A dedicated `net.Listener` (`grpcSrv.Serve(ln)`) gives grpc-go's real `http2Server` where `GracefulStop` drains correctly. And a shared h2c port would let an operator flip their ingress to h2c to serve gRPC, which silently kills the M3 hashserv WebSocket (coder/websocket cannot serve WS over HTTP/2 by any route → 426 → bitbake degrades to `unihash=taskhash`). All three listeners (public, gRPC, metrics) bind before any serves; if one dies all come down. Shutdown order: `public.Shutdown` → `grpc.GracefulStop` (bounded, then `Stop`) → `metrics.Shutdown`.

**sstate misses return 404, never 403.** BitBake retries a 403 as a full-body GET. Never return a 200 with an empty body. This extends to an **unconfigured backend**: a project/kind with no `cache_backends` row 404s — it never mounts a mount point it cannot serve, and never 500s.

**`HEAD` is the sstate hot path, not `GET`.** BitBake fires a `BB_NUMBER_THREADS`-parallel HEAD storm over the whole setscene graph at build start.

**Cache writes ALWAYS require a WRITE-scoped key — reads may not.** `ReadAuthRequired` is per-backend and may be off (open mirror); there is deliberately no `WriteAuthRequired`, because an unauthenticated write is a cache-poisoning vector that must not be a representable state. A write with no/invalid credential is `401 + WWW-Authenticate: Basic realm="bakery"`; authenticated-but-not-write-scoped is `403`. Never a silent accept. A PUT of an existing immutable key is a `200` idempotent no-op (Overwrite is `no` per policy) — not a 409 and not a content swap.

**A Bakery cache credential is one opaque `bkry_` token, NOT an `id:secret` pair.** BitBake speaks only HTTP Basic; `auth.AuthenticateCache` reads the token from the Basic **password field and falls back to the username**, so a client that puts the whole token in either field authenticates. The netrc/env the snippet generator emits therefore carries the token verbatim in both fields — do not "fix" it back into a username+password split; there is no secret half to split off.

**hashserv: one writer goroutine per connection, strictly-ordered responses.** The protocol has no request IDs. A single dropped or reordered response desynchronizes the connection permanently and *silently* — bitbake hangs forever with no error. If two goroutines can write to a connection, the design is already wrong. The client's `Batch` asserts it got exactly as many replies as it sent, so one dropped reply misaligns the list and every subsequent lookup returns the *previous* task's unihash — a wrong sstate object, not an error.

**hashserv auth is denied IN-BAND. NEVER 401 the WebSocket upgrade.** A 401 surfaces in the Python client as `ConnectionError` — and `bb.siggen` **catches it, warns, and sets `unihash = taskhash`**, so the build *completes* with a silently degraded cache (the sstate filename embeds the unihash, so every task the server would have remapped now misses its object). The loud path — the only one that halts the build — is `{"invoke-error":…}` followed by closing the connection. A stock client sends **no `Authorization` header on the upgrade at all**: credentials arrive in-band, in the `auth` RPC, *after* it. So the upgrade must succeed with no credential and deny on the first privileged RPC. (An *unconfigured* backend is different, and 404s: there is nothing there to serve, and degrading is correct.)

**Over WebSocket, the hashserv handshake is ONE MESSAGE PER LINE, with NO NEWLINES.** `protocols/yocto.md` §2.2's newline-terminated form is **stream-transport-only**. The client sends `"OEHASHEQUIV 1.1"`, `"needs-headers: false"`, and an **empty message** as three separate WS frames. A server that waits for a newline-delimited blob waits forever — and so does the build. Same asymmetry in stream mode: `"ok"` (JSON, *quoted*) to **enter**, `ok` (raw, *unquoted*) to **exit**, neither with a newline.

**Verify bitbake claims against the tag the build PINS (`2.8.0`), not branch `2.8`.** They are different trees and the difference already shipped a bug: the 64-hex `UNIHASH_REGEX` is a **2.10+** addition. Scarthgap validates unihashes *not at all* and its own suite reports **40-hex** ones, so enforcing `^[0-9a-f]{64}$` refuses a legitimate client (it failed 11 of 17 gate tests). Bakery accepts `^[0-9a-f]{1,64}$` — hex-only, because the unihash lands in an sstate *filename*.

**hashserv's `get-outhash` accepts a `taskhash` and IGNORES it.** Both lookups behind it key on `(method, outhash)` **alone**, oldest-first; only the *equivalence* query uses the taskhash, and only to **exclude** it. The question is "has **any** task produced this output" — the only form that can discover an equivalence the caller does not already know. Adding `taskhash` to the predicate looks tighter and is silently catastrophic: `report_readonly` then finds an equivalent *never*, so an open mirror serves every request, looks healthy, and delivers **zero** equivalence.

**Parse ByteStream resource names by scanning for the `blobs`/`uploads`/`compressed-blobs` marker.** `instance_name` contains slashes (it's `{org}/{project}`). Never split positionally on `/`.

**Never re-serialize an OCI manifest.** A `json.Marshal` round-trip breaks `Docker-Content-Digest`, and it only reproduces on multi-arch index manifests — i.e. not in your test, and yes in production.

**The OCI upstream fetch requires a verified principal.** No exported way to construct one outside `internal/auth`. Otherwise we are an open relay serving Docker Hub with *our* rate-limit-bearing credentials.

**GC sweeps carry a write barrier, and it has TWO halves.** `created_at < gc_run.started_at` **and** `pg_visible_in_snapshot(live_xid, gc_runs.snapshot)`. The timestamp alone is *not* sufficient and it is not the invariant: a row inserted by a transaction that began before the run but committed after it satisfies the timestamp predicate and must still be spared. `TestGCWriteBarrierSparesAConcurrentBuild` asserts precisely that the timestamp predicate *does* select the row and the sweep spares it anyway. Drop the snapshot half and a build that starts mid-GC loses its freshly-minted unihashes.

**Prometheus labels use slugs, never keys or digests.** Label HTTP metrics on `r.Pattern`, not `r.URL.Path` — the latter creates one time series per sstate object and kills Prometheus inside a single build. Read `r.Pattern` **after** `next.ServeHTTP` returns: `ServeMux` sets it in place, so a middleware outside the mux sees `""` before, labels every request `""`, and still passes a naive smoke test.

**`/metrics` is served on `--metrics-addr` and NOWHERE else.** Its own `http.Server`, its own listener, its own graceful shutdown, loopback by default. It exposes every org and project slug and how many bytes each stores; on the public listener that is the customer list. Both listeners bind before either serves, and if one dies both come down — a live cache with a dead metrics listener is a silently unmonitored server. The assertion that proves it is on the *body*, never the status code: the SPA catch-all answers 200 for everything.

**`/readyz` really pings the pool; `/healthz` does not.** A readyz that returns 200 with a dead database keeps the node in rotation while every request behind it 500s, which is worse than having no readyz. Conflating the two in the other direction makes an orchestrator restart a healthy binary because Postgres blinked.

**Boot takes `pg_try_advisory_lock` and refuses a second instance** unless `--allow-multi-instance`. That refusal is what makes the in-process LRU and route cache sound. It is taken **before** the migrations, so two instances starting together cannot race through the same migration.

**`DEV_LOGIN_ENABLED` is settable only via env var / CLI flag.** No UI or API path may enable it. Defaults off. When off, the endpoint is not registered at all and **404s** — a 403 confirms the endpoint exists and tells a scanner what to come back for.

**Login reconciliation fails closed on an UNREADABLE groups claim — never on an empty one.** "The IdP says this user is in zero groups" and "we could not read this user's groups" are categorically different, and only the first is safe to act on. `groups: []` is ordinary and admissible: under the hybrid model it means *"this user holds only local memberships"*, and it reconciles the OIDC half away. But an **absent** claim, or Azure AD's `_claim_names`/`_claim_sources` overage (which replaces a large groups claim with a pointer at Graph — for real, correctly-configured users), means we do not know: **refuse the login and reconcile nothing. Not one row.** Treating it as "zero groups" NULLs every `oidc_role`, deletes every membership with no local grant, and cascades away that user's project roles and API keys, irreversibly. This holds **whether or not the login gate is enabled** — the gate and the reconciler read the same claim, so disabling the gate must not disable the trap detection. A `[]string` cannot express the difference and is itself the bug; the boundary carries `GroupsPresent`. (M1's rule — *"resolves to no mapped orgs ⇒ refuse"* — is **gone, and must not come back**: a local-only user legitimately maps to zero orgs, and refusing them was the bug. The rule moved; it did not vanish.)

**API-key scope is capped at grant time, and revoked in the same transaction as a role downgrade.** Validating a key is one zero-join index-only probe — deliberately no join onto `project_memberships`, because that would put a second query on the sstate HEAD storm. *Three* things make that join-free grant safe, and it needs all three: the cap, the downgrade revocation, and the **FK cascade** — `org_memberships` DELETE → `project_memberships` (composite FK on `(user_id, org_id)`) → `api_keys`. One DELETE revokes every key a user holds in an org the moment they leave it, with no application code in the loop. Break any of the three and a key outlives the authority it was minted against, silently, for its whole lifetime. An API-key principal never inherits its owner's site admin: a delegation must not become a master key.

**`org_memberships` is ONE ROW per `(user, org)`. The two role sources are COLUMNS (`oidc_role`, `local_role`), never rows.** A `PRIMARY KEY (user_id, org_id, source)` looks like the obvious modelling and is a silent catastrophe: it destroys the uniqueness of `(user_id, org_id)`, which destroys `project_memberships`' composite FK, which destroys the cascade above — and the API keys it was supposed to revoke stay alive. No test you are likely to write catches it. If a change appears to need two rows, that is a design bug, not an implementation detail.

**The reconciler writes ONLY the `oidc_*` columns.** A local grant is not in its keep-set and not in its delete predicate, so a login *cannot* clobber one — structurally, not by convention. `local_role`, `granted_by`, `granted_at` and `project_memberships` are never read and never written on the login path. The moment the reconciler learns to write the local half, "your LDAP group was removed" starts deleting grants a human deliberately made.

**Effective role is `greatest(oidc_role, local_role)`, computed by the DATABASE.** The enums are declared in privilege order (`member < admin < owner`, `user < admin`), so `greatest()` *is* the max-wins rule and there is no application code that can get it wrong. Recomputing it in Go reintroduces exactly the drift the generated column exists to abolish.

**An org membership row is deleted exactly when BOTH sources are gone — and a row justified by neither must not exist.** Deleting it while `local_role` survives revokes keys the user is still entitled to. *Not* deleting it when neither source justifies it suppresses the project-membership cascade and leaves keys alive that should be dead. So the reconciler's delete is guarded (`DELETE ... WHERE local_role IS NULL`), never blind, and a CHECK forbids the both-NULL row outright.

**Org creation grants its creator a LOCAL owner role, in the same transaction as the org.** Without it, a fresh org has no members, so nobody can be given a project role in it, so `CreateAPIKey` refuses with `scope_exceeds_role` — and the console's headline flow (create org → create project → mint key → copy the snippet) dead-ends. That was M1's bug. A crash between the two writes recreates it, which is why it is one transaction and not two.

**`DELETE /orgs/{org}/members/{user}` clears the LOCAL half only, and the response must say when the membership survives.** LDAP owns a claim-derived membership and the API cannot remove it. An admin who removes someone, sees a 200, and reasonably concludes they are gone — while their `oidc_role` still admits them — has been lied to by the API. Membership responses carry provenance (`role`, `oidc_role`, `oidc_group`, `local_role`, `granted_by`, `granted_at`) for the same reason: a local grant that outlives an LDAP revocation must be *visible*, or it is a backdoor.

**`auth.Principal` is unforgeable by construction** — an interface sealed with an unexported method, no exported constructor, no usable zero value. Not a convention; the compiler enforces it (`internal/auth/forgery/` compiles four attacks on purpose and asserts they are rejected).

**Blob paths stay READ COMMITTED.** Under REPEATABLE READ the snapshot is taken at the transaction's *first* statement — the advisory-lock acquisition, before the lock is granted — so the post-lock `SELECT` reads a stale world and the `FOR UPDATE` + `refcount = 0` recheck protects nothing. A `pgx.TxOptions{IsoLevel: RepeatableRead}` anywhere on a blob path is a silent correctness bug.

**The LRU caches NEGATIVE results and is sharded.** On the first build against an empty cache *every* HEAD is a miss, so a positive-only cache sends the entire setscene graph to Postgres on every build — and no test that pre-populates the repo will ever show it. And a single-mutex LRU gets *slower* as parallelism rises (measured 170ns@8 → 246ns@64), which is exactly backwards under a `BB_NUMBER_THREADS` HEAD storm.

**`ServeMux` panics you will meet when the cache routes land.** `/cache/{org}/{project}/{kind}/{key}` alongside `/cache/{org}/{project}/sstate/{path...}` panics at registration — register `ac`/`cas` as **literal** segments. A method-less `/v2/{org}/{project}/` alongside `GET /` panics too. This is why the SPA is registered as a method-less `/` (it does its own method check) and `/api/v1/` mounts cleanly beside it.

## Frontend: how it's structured

The Claude Design handoff has **landed**. The design system is ported into the SvelteKit app under `web/` — build against it, don't redesign it.

- **Tailwind v4, CSS-first.** No `tailwind.config.js`. Tokens are CSS variables in `web/src/lib/styles/tokens/` (colors dark on `:root`, light under `[data-theme="light"]`), mapped onto Tailwind's `--color-*`/`--radius-*`/font namespaces via an `@theme` block in `web/src/app.css`. Utilities like `bg-bg-1 text-text-2 border-border-0 rounded-2` resolve to those vars, so flipping `<html data-theme>` recolors live with no rebuild. **No raw hex in markup**; never use `dark:` variants.
- **Theming** is `web/src/lib/theme.ts` (`dark|light|system`, persisted to `localStorage["bakery-theme"]`, default `system`), plus a no-FOUC script in `app.html`. The ConsoleNav footer toggle and User Settings "Appearance" control drive the same store — one source of truth.
- **Component library** in `web/src/lib/components/<group>/` (buttons, inputs, badges, table, navigation, feedback, content, data). These rebuild the `.bk-*` recipes as `.svelte` components using token utilities. Reuse them; do not ship `.bk-*` classes or reintroduce inline primitives. Status is a typographic glyph + color, never an icon: `●` hit, `✕` miss, `▲` stale, `○` idle, `∅` empty.
- **Routes**: a `(console)` group layout renders `<ConsoleNav/>` + the page; `/login` is full-screen outside it; `/` redirects to `/overview`. adapter-static SPA (`ssr=false`), built to `web/dist`, embedded via `//go:embed all:dist`.
- **All screens are built with mock data — API wiring is still pending.** When you add a backend, replace the in-component mock arrays with real data; the visual layer stays as-is. `web/FOUNDATION.md` is the binding contract for the utility vocabulary and component prop APIs.
- Fonts still load via a Google Fonts `@import`; production should self-host the woff2 for the offline embedded console (not a blocker).

Voice/fidelity is fixed by the design system: sentence case, terse and technical, no emoji, no exclamation points, second person; metrics in tabular numerals, mono for hashes/keys/config.

## Domain model

`Organization` → `Project` → `Cache Backend` (sstate | downloads | hashserv | bazel | oci). Users have org and project roles: an **org** role is hybrid (OIDC group **or** local grant, effective = the greater), a **project** role is in-app only. API keys are project-scoped and per-user, read/write-scoped, shown once.

**sstate and hashserv are hard-coupled**: the sstate object filename embeds the *unihash*, so the two must be scoped, retained, and garbage-collected together. The unihash is the GC root; sstate objects are reachable-from-unihash. Always sweep hashserv before sstate.

## Addressing

```
/cache/{org}/{project}/sstate/{path...}
/cache/{org}/{project}/downloads/{basename}
/cache/{org}/{project}/hashserv                 (wss upgrade)
/cache/{org}/{project}/{ac,cas}/{hex}
/cache/{org}/{project}/docker/v2/...?ns=        (containerd)
/v2/{org}/{project}/...?ns=                     (BuildKit — prefix lands after /v2)
  gRPC REAPI: project comes from instance_name = "{org}/{project}"
/api/v1/...  /healthz  /readyz  /  (SPA)
```

Reserved slugs (forbid at org/project creation): `blobs`, `uploads`, `actions`, `actionResults`, `operations`, `capabilities`, `compressed-blobs`, `ac`, `cas`, `v2`, `api`, `cache`.
