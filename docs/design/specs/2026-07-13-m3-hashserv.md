# M3 — Yocto hash equivalence server (hashserv)

Status: approved, in implementation. Supersedes the `hashserv` lines in DESIGN.md §"Yocto hashserv"
and §"Auth model" that predate the M3 verification pass.

Everything below marked **[SRC]** was verified by reading bitbake 2.8 (Scarthgap) source, not docs.

---

## 1. What a build actually calls

Grepped `lib/bb/**` (the build path) for every method on the hashserv client. **[SRC]**

| RPC | Called by a real build? |
|---|---|
| `ping`, `get`, `get-stream`, `exists-stream`, `report`, `report-equiv`, `auth` | **yes**, every build |
| `get-outhash` | not from `lib/bb`, but it is the upstream-chaining primitive |
| `remove`, `gc-mark`, `gc-mark-stream`, `gc-sweep`, `gc-status`, `clean-unused`, `get-db-usage`, `get-db-query-columns` | **no** — zero call sites |
| `new-user`, `set-user-perms`, `become-user`, `get-user`, `get-all-users`, `delete-user`, `refresh-token` | **no** — zero call sites |
| `get-stats`, `reset-stats`, `backfill-wait` | no (`backfill-wait` is a CI determinism hook) |

The bottom groups exist for the `bitbake-hashclient` operator CLI and for upstream's own test
suite. They are not load-bearing for bitbake. This is the finding that sets M3's scope: we do not
warp the auth model to serve RPCs no build issues.

### Scope decision

**Implement:** `ping`, `auth`, `get`, `get-stream`, `exists-stream`, `report`, `report-equiv`,
`get-outhash`, `remove`, `backfill-wait`.

**Deliberately NOT implemented** (an unrecognized command is logged and the connection dropped,
which is exactly what upstream does — `ClientError` is never sent to the client **[SRC]**
`serv.py`):

- **The GC RPCs.** Bakery does GC **in-process** in M6, against the database, with the two-half
  write barrier (`created_at < gc_run.started_at` **and** `pg_visible_in_snapshot(...)`).
  Implementing upstream's `gc_mark`/`config['gc-mark']` mark-and-sweep would put a *second,
  competing* GC mechanism on the same table, reachable by cache clients, for zero production
  benefit. It is a footgun aimed squarely at M6. There is therefore **no `gc_mark` column and no
  `config` table.**
- **The user-admin RPCs.** Credentials are minted by the Bakery API. A cache client never mints a
  credential — that rule is not negotiable and these RPCs exist only to break it.

`remove` **is** implemented: it is project-scoped, requires a write-scoped key, and is how the
conformance harness resets between tests. An operator legitimately wants "purge this project's hash
equivalence".

## 2. Wire protocol — the four traps

### 2.1 Auth is denied IN-BAND. Never 401 the upgrade.

This inverts what DESIGN.md originally said. Chain, verified **[SRC]**:

401 at the upgrade → Python `websockets` raises `InvalidStatus` → `asyncrpc/client.py` re-raises as
`ConnectionError` → `_send_wrapper` retries 3× then re-raises (`client.py:144-155`) →
**`siggen.py:851` and `siggen.py:894` catch `ConnectionError`, call `bb.warn()`, and continue.**
The build *completes*, with `unihash = taskhash`. Since the sstate object filename embeds the
unihash, every task whose unihash the server would have remapped now misses its stored object. The
cache silently degrades. Four hours into someone's build.

The loud path is `{"invoke-error": {"message": …}}` followed by closing the connection.
`InvokeError` is in no retry tuple and has no `except` on the build path, so it propagates and
**halts** the build. This is how upstream's own `handle_auth` → `fail_auth()` denies.

So:

| Condition | Response |
|---|---|
| No `cache_backends` row for `{org}/{project}/hashserv` | **404 at the upgrade.** Nothing to serve; matches the sstate unconfigured-backend rule. Degrading is correct here. |
| Credential absent/invalid/insufficient | **In-band `invoke-error` + close.** Loud. Halts the build. |

Never a 401, never a 403, on the upgrade.

### 2.2 A stock client sends NO `Authorization` header on the upgrade.

**[SRC]** `asyncrpc/client.py:86-90` — `websockets.connect(uri, ping_interval=None)`, no headers
argument. Credentials travel in-band via the `auth` RPC, driven by
`hashserv/client.py:89-98 setup_connection`. Gating the upgrade on Basic auth therefore rejects the
connection *before the client has had any chance to present a credential*.

### 2.3 Over WebSocket the handshake is ONE MESSAGE PER LINE, with no newlines.

**This is not in `protocols/yocto.md`, which documents the handshake as "each line
`\n`-terminated" without qualifying that as stream-only.** Building the WS path to that spec hangs
bitbake forever.

`setup_connection` sends the handshake through the transport-polymorphic `send()` **[SRC]**
(`client.py:99-109`):

- `StreamConnection.send(msg)` → `writer.write(msg + "\n")` — newline-terminated lines.
- `WebsocketConnection.send(msg)` → `socket.send(msg)` — **one WebSocket text message, no newline.**

So over WebSocket the client sends, as *separate WS messages*:

```
msg 1:  OEHASHEQUIV 1.1
msg 2:  needs-headers: false
msg 3:  <empty message>          <- terminates headers
```

and the server reads them with one `recv()` per message. The same asymmetry governs stream mode:
`send_message("ok")` → the message `"ok"` (JSON, **with quotes**) to *enter*; `send("ok")` → the
message `ok` (raw, **no quotes**) to *exit*. No trailing newlines anywhere on the WS path.

Accept protocol version `> (1,0) and <= (1,1)` — i.e. `1.1` only. Reject by closing with no
response. **[SRC]** `server.py:300`.

### 2.4 The read limit is mandatory.

`coder/websocket` defaults to a **32 768 byte** read limit. Upstream's `test_huge_message` reports
an `outhash_siginfo` of `"0" * (max_chunk * 4)` = **131 072 bytes** **[SRC]** (`tests.py:279`), and
WebSocket messages are **not** chunked — chunking is stream-transport-only — so it arrives as one
frame and the connection dies with `StatusMessageTooBig`. The collision with bitbake's
`DEFAULT_MAX_CHUNK` (also 32 KiB) is a coincidence that makes the default *look* safe.

`conn.SetReadLimit(hashserv.MaxMessageBytes)` where `MaxMessageBytes = 1 << 20` (1 MiB), chosen to
mirror Python `websockets`' own default `max_size` in both directions: a response larger than that
would be rejected by the *client*, so accepting a larger request would let us store a siginfo we can
never serve back.

### 2.5 One writer goroutine per connection.

The protocol has no request IDs. Responses are strictly in-order, one per request. A single dropped
or reordered response desynchronizes the connection permanently and *silently*; bitbake then hangs
forever with no error. Exactly one goroutine may write to a connection. Upstream's `test_stress`
(100 concurrent clients × 1000 lookups **[SRC]** `tests.py`) is the test that catches a violation.

Also: `ping_interval=None` on both client and server **[SRC]** — bitbake disables WebSocket pings.
The server must not depend on them, and must not apply an idle write/read deadline that would reap a
long-lived connection between setscene bursts.

## 3. Transport

**WebSocket only.** Bakery serves one HTTPS port; the shared mux gives hashserv the same TLS,
ingress and routing as everything else. Raw TCP and unix-socket transports are **deferred** — they
carry no TLS and no auth story, and nothing in Bakery's deployment reaches them.

This is a free risk reduction: the chunked stream framing (`chunk-stream` sentinel, 32 KiB chunks,
bare-line terminator) that DESIGN.md calls the highest-risk file **exists only on the raw-TCP
path**. Not implementing TCP means never writing it.

## 4. Permissions

hashserv's permission vocabulary mapped onto Bakery's. An API-key principal is deliberately hollow —
`CanAdminProject` is hard-false for `MethodAPIKey` — so only two questions can ever be true of a
cache credential.

| hashserv perm | Bakery source | Grants |
|---|---|---|
| `@read` | `CanReadProject`, **or anonymous when `read_auth_required = false`** | `get`, `get-stream`, `exists-stream`, `get-outhash`, `ping` |
| `@report` | `CanWriteProject` (write-scoped key) | `report`, `report-equiv` write path |
| `@db-admin` | `CanWriteProject` (write-scoped key) | `remove` only — project-scoped |
| `@user-admin` | **unreachable** | nothing; the RPCs do not exist |

`auth` accepts the opaque `bkry_` token from **either** the `token` field or the `username` field,
mirroring `AuthenticateCache`'s Basic password-then-username fallback. There is no `id:secret` pair
to split.

**Anonymous `report`.** When `read_auth_required = false` an anonymous client holds `@read` but not
`@report`. Upstream's `handle_report` then takes the `report_readonly` path: it looks the outhash up
and returns the stored unihash if known, else echoes the client's own unihash back, and **never
writes** **[SRC]** (`server.py:514-515`). This is upstream's documented read-only mode and is the
right behavior for an open mirror — but it is a *silent* non-write, so it is metered
(`bakery_hashserv_reports_dropped_total`) and logged, not swallowed.

Writes still always require a key. There is no anon-perms knob and no unauthenticated write path.

## 5. Schema

Migration `000010_hashserv`. Two tables, both scoped by `backend_id` — the multi-tenancy boundary.

Both tables carry **both halves of the M6 GC write barrier** — `created_at` *and* `live_xid`.
The timestamp alone is not the invariant and never was: a row inserted by a transaction that began
before the GC run but committed after it satisfies `created_at < gc_run.started_at` and must still be
spared, which is what `pg_visible_in_snapshot(live_xid, gc_runs.snapshot)` is for. Without
`live_xid`, M6's hashserv sweep is *structurally incapable* of expressing the barrier — and since the
unihash is the sstate GC root, a build running during a GC would lose its freshly-minted unihashes.
It lands with the table because retrofitting a barrier onto a table already accumulating rows is not
a migration, it is an outage.

```sql
CREATE TABLE hashserv_unihashes (
    backend_id  bigint      NOT NULL REFERENCES cache_backends (id) ON DELETE RESTRICT,
    method      text        NOT NULL,
    taskhash    text        NOT NULL,
    unihash     text        NOT NULL,
    -- BOTH halves of the M6 GC write barrier. NOT upstream's gc_mark -- see §1.
    created_at  timestamptz NOT NULL DEFAULT now(),
    live_xid    bigint      NOT NULL DEFAULT (pg_current_xact_id()::text::bigint),
    PRIMARY KEY (backend_id, method, taskhash)
);
CREATE INDEX hashserv_unihashes_unihash ON hashserv_unihashes (backend_id, unihash);

CREATE TABLE hashserv_outhashes (
    backend_id      bigint      NOT NULL REFERENCES cache_backends (id) ON DELETE RESTRICT,
    method          text        NOT NULL,
    taskhash        text        NOT NULL,
    outhash         text        NOT NULL,
    created_at      timestamptz NOT NULL DEFAULT now(),
    owner           text,
    pn              text,
    pv              text,
    pr              text,
    task            text,
    outhash_siginfo text,
    PRIMARY KEY (backend_id, method, taskhash, outhash)
);
-- The equivalence query. Covers (backend_id, method, outhash) ORDER BY created_at.
CREATE INDEX hashserv_outhashes_equiv
    ON hashserv_outhashes (backend_id, method, outhash, created_at);
```

`(backend_id, method, taskhash) → unihash` is **write-once**: `INSERT ... ON CONFLICT DO NOTHING`.
Once a taskhash has a unihash it never changes.

`outhash_siginfo` stays a Postgres `text` column, as upstream does. It is TOASTed automatically, it
is only ever read by `get-outhash`/`get(all=true)`, and routing it through `blob.Service` would
couple the unihash — the GC *root* — to the refcounted object store it is supposed to root.

## 6. The equivalence algorithm

Reproduce `handle_report` exactly **[SRC]** (`server.py:467+`, bitbake **2.8.0**):

1. Validate `unihash` against **`^[0-9a-f]{1,64}$`** — *not* `^[0-9a-f]{64}$`.

   **This step as originally specified was wrong, and the conformance gate caught it.** The
   64-hex `UNIHASH_REGEX`/`is_valid_unihash` is a **2.10+ addition**; bitbake 2.8 — Scarthgap, our
   floor — does not validate unihashes at all, and its own suite reports **40-hex** ones
   (`create_test_hash` → `f46d3fbb439bd9b921095da657a4de906510d2cd`). Enforcing 64 refused a
   legitimate client and failed 11 of the 17 gate tests. We accept the union of every supported
   release, and keep it hex-only and length-bounded because the unihash is embedded in the sstate
   object filename.

   *Method note, recorded because it nearly cost a day:* the original finding came from reading
   git branch `2.8`, which tracks later 2.8.x, while the repo pins **tag `2.8.0`**. Verify against
   the tree the build actually pins.
2. If the caller lacks `@report` → `report_readonly`: look up, return stored-or-echoed, **no write**.
3. `INSERT` the outhash row `ON CONFLICT DO NOTHING`. **Only if a NEW row was created:**
   a. Find a *different* taskhash with the same `(backend_id, method, outhash)`, **oldest first** —
      that row's unihash wins. This is the equivalence.
   b. Else mint the reported unihash — but if an upstream is configured, ask it `get-outhash`
      first and adopt upstream's unihash if it has one, so we cannot diverge from upstream for an
      output upstream already knows.
   c. `INSERT` the unihash row `ON CONFLICT DO NOTHING`.
4. **Always re-read** `(method, taskhash) → unihash` and return that; another writer may have won
   the race. Fall back to the client's unihash only if the row is somehow absent.

Equivalence is computed **only when the outhash row is new** — a duplicate report is a cheap lookup.
The chosen unihash is the one on the **oldest** matching outhash row, which is what makes unihashes
stable over time.

Blob paths stay `READ COMMITTED`; the same applies here. Do not reach for `REPEATABLE READ`.

## 7. Upstream chaining

In scope for M3, **disableable**. Configured per backend via `cache_backends.config`:

```json
{ "upstream": "wss://hashserv.yoctoproject.org/ws" }
```

Absent or empty ⇒ chaining off (the default). A server-wide kill switch
`--hashserv-disable-upstream` (env `HASHSERV_DISABLE_UPSTREAM`) overrides every backend, for
the day the public server is down and it is showing up in customer builds.

Semantics **[SRC]** (`server.py`):

- `get` / `get-outhash` miss ⇒ synchronous upstream query, **write through** the result.
- `get-stream` miss ⇒ query upstream; on a hit, **enqueue `(method, taskhash)` for backfill and
  return the unihash immediately**. The write never blocks the streaming hot path.
- `exists-stream` miss ⇒ ask upstream, no backfill.
- `report` minting a brand-new unihash ⇒ ask upstream `get-outhash` first, adopt if known.
- `backfill-wait` ⇒ drain the queue, return the depth seen at entry. This is what makes the
  conformance run deterministic.

A background backfill worker drains the queue and validates `is_valid_unihash` before inserting.

**Bakery pools upstream connections; the reference implementation does not.** Upstream opens one
upstream connection *per downstream client* **[SRC]** (`ServerClient.process_requests`). With a
`BB_NUMBER_THREADS`-wide build that is a connection storm aimed at a third party. We keep a small
bounded pool per backend.

This requires a Go hashserv **client**, which lives in `upstream.go` and is independently useful.

## 8. Package layout

```
internal/cache/hashserv/
  protocol.go   RPC names, request/response shapes, unihash validation. Pure.
  framing.go    WS handshake (message-per-line), normal mode, stream mode. Pure. Exhaustive tests.
  conn.go       Per-connection state machine. ONE writer goroutine. Read limit.
  handlers.go   The RPC handlers.
  store.go      DB access + the equivalence algorithm.
  upstream.go   Go hashserv client + backfill worker + connection pool.
  perms.go      auth.Principal -> hashserv permissions.
  backend.go    cache.StreamBackend: Kind() / Register() / ServeStream().
```

`cache.StreamBackend` already exists (`internal/cache/backend.go:203`) and was written for exactly
this. hashserv does **not** route through `blob.Service` — it is the one backend that doesn't — so
it owns its metrics (`Deps.Metrics`, per the `Deps` doc comment).

`internal/auth` gains one exported method, `AuthenticateToken(ctx, token) (Principal, error)` — a
thin wrapper over the existing `authenticateKey`, because the in-band `auth` RPC has no
`*http.Request` to hand to `AuthenticateCache`. It must be added to the allow-list in
`TestPrincipalHasNoExportedConstructor` with a written justification.

## 9. The CI gate

DESIGN.md calls this non-negotiable. It has **two halves, and upstream only gives us one.**

**Half 1 — upstream's own suite, external mode.** `BB_TEST_HASHSERV=ws://127.0.0.1:PORT/cache/{org}/{project}/hashserv`
driving `hashserv.tests.TestHashEquivalenceExternalServer`. The address is passed verbatim to
`websockets.connect` **[SRC]** (`parse_address`, `client.py:27-40`), so the multi-tenant path
survives.

We run it authenticated (a real `bkry_` write key in `BB_TEST_HASHSERV_USERNAME`/`PASSWORD`) and
filter to the tests that exercise what a build does:

- **Run:** `test_create_hash`, `test_create_equivalent`, `test_duplicate_taskhash`,
  `test_remove_*` (4), `test_huge_message`, `test_stress`, `test_unihash_exsits` *(sic, upstream's
  typo)*, `test_diverging_report_race`, `test_diverging_report_reverse_race`,
  `test_client_pool_get_unihashes`, `test_client_pool_unihash_exists`.
- **Excluded, by name, with a recorded reason:** the 15 `test_auth_*` (user-admin surface we
  deliberately refuse), the 4 `test_gc*` + `test_clean_unused` + `test_get_db_usage` +
  `test_get_db_query_columns` (GC RPCs we deliberately refuse — Bakery GCs in-process), and the 3
  that call `start_server()` and skip themselves in external mode.

The gate asserts **exact** `ran` / `failures=0` / `errors=0` / `skipped` counts, so an exclusion that
silently grows — or a suite that skips everything because an env var came through empty and reports a
triumphant green — is caught. `get_env` skipTests on an empty string too **[SRC]**.

**Half 2 — the real `bitbake-hashclient`, which upstream's external suite never runs.** All 27
`run_hashclient` call sites live in `TestHashEquivalenceClient`, which spawns its *own local*
`unix://` python server; `TestHashEquivalenceExternalServer` does not inherit them **[SRC]**. So
"the real hashclient against our server" is a **Bakery-authored driver** — the analogue of
`test/conformance/testdata/phase_a.py` — shelling out to `build/bitbake/bin/bitbake-hashclient
--address ws://... -n <subcmd>` and asserting on machine-parsable output.

Pass `-n/--no-netrc`: `bitbake-hashclient` reads `~/.netrc` keyed on the **full ws:// URL** **[SRC]**
(which independently corroborates the netrc gotcha in DESIGN.md), and a stray netrc on a runner must
not perturb the run.

### Harness constraints

- Lives under `./test/hashserv/`, **not** `./internal/...` — `just test-db` globs `./internal/...`
  and fails on any SKIP, and this suite legitimately skips on a laptop with no bitbake checkout.
  Same reason `./test/conformance` lives where it does. But `just race`/`just coverage` glob `./...`,
  so it must skip **cheaply and early** — before `dbtest.New(t)` and before booting a server.
- **Pin `websockets`.** bitbake 2.8 calls the legacy `websockets.connect(...)` top-level API, which
  was deprecated and then removed in websockets 14. An unpinned `pip install websockets` yields a red
  CI that is not a Bakery bug. Pin it beside `bb_tag := "2.8.0"`, as a matched set.
- `pip install` on ubuntu-latest is PEP 668 externally-managed — use `actions/setup-python`, and make
  sure the `python3` the Go harness `exec.Command`s is the same interpreter.
- The new job must be added to `image.needs` or it gates nothing.

## 10. Metrics

`bakery_hashserv_*`, emitted from `Deps.Metrics` (hashserv does not route through `blob.Service`, so
it gets none of the headline series for free). Labels are resolved slugs from a `Route`, never keys
or hashes.

- `bakery_hashserv_connections{org,project}` — gauge, live connections.
- `bakery_hashserv_rpcs_total{org,project,method,result}` — `result` = ok|error|denied.
- `bakery_hashserv_stream_lines_total{org,project,stream}` — the hot path's real volume.
- `bakery_hashserv_equivalences_total{org,project}` — how often we actually remapped a unihash. This
  is the number that says whether the whole system is earning its keep.
- `bakery_hashserv_reports_dropped_total{org,project,reason}` — the silent non-writes, made loud.
- `bakery_hashserv_upstream_total{org,project,op,result}` and `bakery_hashserv_backfill_queue{org,project}`.
